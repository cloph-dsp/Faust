# Fix GUI Lag with Large FFT Sizes

## Problem
When FFT size is large (e.g., 8192 or 16384), the GUI becomes laggy because:
1. Large FFT computation blocks the audio thread
2. Visualizer updates happen every hop (too frequent for large FFTs)
3. `GetUI()` is called on audio thread causing contention

## Solutions (in order of effectiveness)

### Option A: Reduce Visualizer Update Rate (Recommended)
- Add a frame counter to only update visualizer every N hops
- Scale N based on FFT size (larger FFT = less frequent updates)
- Trade-off: Visualizer is slightly less smooth but GUI remains responsive

### Option B: Separate FFT for Visualizer
- Use full FFT for audio processing
- Use smaller FFT (e.g., 2048) just for the visualizer display
- Trade-off: Visualizer shows less frequency resolution than actual processing

### Option C: Interpolation/Smoothing
- Keep last computed spectrum data
- Interpolate between frames instead of computing fresh every time
- Trade-off: Visualizer shows smoothed/interpolated data

### Option D: Async UI Update
- Copy spectrum data to atomic/shared buffer
- Let UI thread poll the data asynchronously
- Trade-off: More complex, requires thread safety

## Implementation (Option A)

Add to `ProcessHop()` in `RussetNoise.cpp`:

```cpp
// Skip visualizer updates for large FFTs to maintain UI responsiveness
static int visUpdateCounter = 0;
int visSkipFrames = std::max(1, mFFTSize / 2048); // Skip more frames for larger FFTs
visUpdateCounter++;
bool shouldUpdateVis = (visUpdateCounter >= visSkipFrames);
if (shouldUpdateVis) {
    visUpdateCounter = 0;
}

// ... compute beforeSpec, afterSpec, referenceSpec, freqSpec ...

// Update spectrum visualizer control (conditional)
if (shouldUpdateVis) {
    if (IGraphics* g = GetUI()) {
        for (int i = 0; i < g->NControls(); ++i) {
            if (auto* vis = dynamic_cast<SpectrumVisualizerControl*>(g->GetControl(i))) {
                vis->SetData(beforeSpec, afterSpec, referenceSpec, freqSpec, visBands);
                break;
            }
        }
    }
}
```

This provides:
- 8192 FFT: update every 4 hops (~11 Hz at 44.1kHz)
- 16384 FFT: update every 8 hops (~5.5 Hz at 44.1kHz)
- 2048 FFT: update every 1 hop (~43 Hz at 44.1kHz)

## Benefits
- Simple implementation, no threading complexity
- Audio processing remains unchanged (full FFT quality)
- Visualizer remains functional but updates less frequently
- GUI stays responsive even with large FFTs
